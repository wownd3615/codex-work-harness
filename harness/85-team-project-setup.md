# Team Project Setup

팀 단위 프로젝트에서는 개인 하네스를 그대로 쓰지 않는다. 팀 합의, 저장소 규칙, 리뷰 흐름을 함께 세팅한다.

## 저장소 구조
프로젝트 루트에 둔다.

```text
AGENTS.md
harness/
templates/
scripts/
docs/
```

팀 전용 규칙은 `TEAM_RULES.md` 또는 `docs/team-rules.md`에 둔다.

## 필수 합의
- 브랜치 전략: `main`, `develop`, feature branch 중 선택
- PR 규칙: 누가 리뷰하고 언제 머지하는지
- 코드 스타일: formatter/linter/test 명령
- 보안 규칙: secret, env, 권한, 개인정보 처리
- 이슈/작업 관리: GitHub Issues, Linear, Notion 등
- 배포 권한: 누가 배포하고 롤백하는지

## AI 사용 규칙
- AI가 만든 코드는 사람이 PR에서 리뷰한다.
- AI는 테스트하지 않은 내용을 통과했다고 말하지 않는다.
- AI가 대규모 변경을 제안하면 먼저 범위와 위험을 문서화한다.
- AI는 secret, token, 개인정보를 코드/문서에 남기지 않는다.
- AI는 기존 팀 규칙과 충돌하면 팀 규칙을 우선한다.

## 역할
- Owner: 최종 의사결정, 범위/비용/일정
- Tech Lead: 구조, 코드 품질, 리뷰 기준
- Product: 요구사항, MVP, 우선순위
- Design/UX: 화면 흐름, 사용성
- Security: 인증, 권한, 개인정보, 배포 위험
- QA: 테스트, 회귀 확인, 릴리즈 체크

작은 팀에서는 한 사람이 여러 역할을 맡아도 된다. 역할은 사람 수가 아니라 책임을 분명히 하기 위한 것이다.

## 팀용 시작 절차
1. `AGENTS.md`와 하네스 적용 여부 합의
2. 프로젝트 레벨 판단: Starter/Dynamic/Enterprise
3. 외주/내부개발/혼합 개발 여부 판단
4. MVP와 제외 범위 합의
5. 보안/사업성/UI/UX 게이트 적용 기준 합의
6. PR, 테스트, 배포 규칙 확정
7. 첫 작업은 작은 기능 하나로 하네스가 잘 맞는지 검증

## PR 체크리스트
- 요청 범위만 변경했는가
- 코드가 최소한으로 짧은가
- 필요 없어진 코드를 삭제했는가
- 테스트/빌드/브라우저 검증 결과가 있는가
- 보안/개인정보 위험이 없는가
- UI/UX 상태가 깨지지 않는가
- 불확실한 점이 PR 설명에 적혀 있는가
