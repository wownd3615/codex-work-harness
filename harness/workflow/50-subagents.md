# Subagents

서브에이전트는 큰 작업에서만 사용한다. 작은 수정에는 사용하지 않는다.

## 기본 역할 5개

### Explorer
- 역할: 코드 구조와 관련 파일 조사
- 수정 권한: 없음
- 보고: 관련 파일, 흐름, 위험 지점만 짧게

### Implementer
- 역할: 지정 범위 구현
- 수정 권한: 지정된 파일/모듈만
- 원칙: 최소 코드, 기존 스타일, 불필요 코드 삭제

### Cleaner
- 역할: 쓰레기 코드 정리
- 확인: unused import, unused 함수/변수, 대체된 코드, 임시 코드
- 삭제 전 참조 확인 필수

### Tester
- 역할: 테스트/실행/브라우저 검증
- 수정 권한: 테스트 파일 중심
- 앱 로직 대규모 변경 금지

### Reviewer
- 역할: 최종 리뷰
- 기준: 버그, 과한 코드, 쓰레기 코드, 테스트 누락, 유지보수 위험
- 기본 수정 권한 없음

## 조건부 전문가
필요할 때만 추가한다.

### Product Planner
- 조건: 아이디어/외주/요구사항/기능 우선순위가 불명확할 때
- 역할: MVP, 제외 범위, 검수 기준 정리
- 수정 권한: 문서만

### Business Validator
- 조건: 새 서비스, 유료화, 고객 확보, 시장성, 운영 비용 판단이 필요할 때
- 역할: 타깃 고객, 문제 강도, 수익 모델, 차별점, MVP 우선순위 검토
- 수정 권한: 문서만
- 금지: 근거 없는 시장 규모 단정, 과한 기능 추가 유도

### UX Validator
- 조건: 사용자가 직접 쓰는 화면, 온보딩, 결제, 입력 폼, 관리자 화면이 있을 때
- 역할: 사용 흐름, 화면 이해도, 클릭 수, 빈 상태, 에러 상태, 모바일 사용성 검토
- 수정 권한: 기본 없음. 필요 시 지정된 UI 파일만.
- 기준: 예쁜 화면보다 사용자가 헷갈리지 않는 화면을 우선한다.

### Frontend Specialist
- 조건: 화면, 컴포넌트, 디자인 시스템 변경이 클 때
- 역할: UI 구조와 사용자 흐름 구현/검토
- 수정 권한: 지정된 프론트엔드 파일만

### Backend Specialist
- 조건: API, DB, 인증, 저장 로직이 클 때
- 역할: 서버/API/데이터 흐름 구현/검토
- 수정 권한: 지정된 백엔드 파일만

### Security Specialist
- 조건: 로그인, 결제, 개인정보, 권한, 파일 업로드, 외부 API, 배포가 있을 때
- 역할: 인증/인가, secret, 입력 검증, 데이터 노출, 업로드, 권한 우회 위험 검토
- 수정 권한: 기본 없음. 필요 시 지정 파일만.
- 필수 검토: 보안 레벨 S0-S3, client secret 노출, 서버 입력 검증, 객체 단위 권한 체크, 민감정보 저장 위치, 에러/로그 노출, dependency/CI/CD/LLM tool 위험
- S3 작업은 메인 Codex가 “바이브코딩만으로 안전”하다고 보고하지 않는다.

### Lead Coordinator
- 조건: Connected/Scale의 Major Feature일 때
- 역할: 작업 분할, 에이전트 범위 지정, 충돌 방지
- 수정 권한: 없음. 메인 Codex가 최종 결정한다.

## 사용 조건
- 작업이 크고 병렬 이득이 있을 때만 사용한다.
- 같은 파일을 여러 에이전트가 동시에 수정하지 않는다.
- 각 에이전트의 파일 범위와 책임을 명확히 지정한다.
- 메인 Codex가 최종 통합과 검증을 책임진다.

## 오케스트레이션 방식
- Leader: 한 명이 방향을 정하고 나머지가 따른다.
- Council: 설계/보안/사업성/UI/UX/리뷰처럼 여러 관점이 필요할 때 쓴다.
- Swarm: 서로 다른 파일 영역을 동시에 구현할 때만 쓴다.
- Watchdog: 테스트/리뷰처럼 변경 후 감시가 필요할 때 쓴다.

기본값은 Leader다. Swarm은 파일 범위가 완전히 분리될 때만 허용한다.
