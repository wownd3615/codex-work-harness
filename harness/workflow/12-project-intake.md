# Project Intake

프로젝트 시작 전에는 필요한 정보를 먼저 인터뷰한다.

목표는 질문을 많이 하는 것이 아니라, 나중에 갈아엎을 가능성을 줄이는 것이다.

## 인터뷰 순서
프로젝트 시작 시 아래 순서로 확인한다.

1. 목적
2. 사용자
3. 문제
4. 범위
5. 우선순위
6. 성공 기준
7. 리스크

## 필수 질문 7개
아래를 짧게 확인한다.

1. 무엇을 만들고 싶은가?
2. 누가 사용할 것인가?
3. 지금 가장 불편한 문제는 무엇인가?
4. 꼭 필요한 핵심 기능 3개는 무엇인가?
5. 이번 버전에서 만들지 않을 것은 무엇인가?
6. 어떤 파일/데이터/시스템을 사용해야 하는가?
7. 성공했다고 판단할 기준은 무엇인가?

## 우선순위 분류
기능은 아래 네 단계로 나눈다.

| 분류 | 의미 | 처리 |
|---|---|---|
| Must | 이번 버전에 반드시 필요 | MVP에 포함 |
| Should | 중요하지만 없어도 첫 검증 가능 | 여유가 있으면 포함 |
| Could | 있으면 좋음 | 다음 버전 후보 |
| Won't | 이번에는 만들지 않음 | 제외 범위에 기록 |

## 인터뷰 결과물
인터뷰 후 바로 코딩하지 말고 먼저 아래를 정리한다.

- 한 문장 목표
- 핵심 사용자
- 해결할 문제
- Must 기능 3개 이하
- 이번에 만들지 않을 것
- 필요한 파일/데이터/시스템
- 성공 기준
- 주요 리스크

## 상황별 추가 질문

### 외주
- 납품물은 무엇인가?
- 검수 기준은 무엇인가?
- 유지보수와 수정 범위는 어디까지인가?

### 팀 프로젝트
- 브랜치/PR/리뷰 규칙은 무엇인가?
- 누가 최종 결정권자인가?
- 테스트와 배포 책임자는 누구인가?

### 업무 자동화
- 입력 파일은 어디에 있는가?
- 결과물은 어디에 저장해야 하는가?
- 삭제/이동/전송 같은 위험 작업이 있는가?
- 샘플 파일로 먼저 테스트할 수 있는가?

### 웹앱/프로그램
- 로그인/권한이 필요한가?
- 저장해야 할 데이터는 무엇인가?
- 내부용인가 외부 공개용인가?

### CAD LSP
- 대상 CAD는 AutoCAD, BricsCAD, ZWCAD 중 무엇인가?
- 기존 LSP 파일 인코딩은 무엇인가?
- 도면을 수정하는 명령인가, 조회만 하는 명령인가?

## 진행 규칙
- 사용자가 이미 말한 내용은 다시 묻지 않는다.
- 모르면 합리적으로 가정하되, 위험한 결정은 확인한다.
- 질문은 한 번에 너무 많이 하지 않는다.
- 간단한 프로젝트는 필수 질문을 요약해서 3개 이하로 줄인다.
- 답변 후 MVP와 제외 범위를 먼저 정리한다.
- Must 기능이 3개를 넘으면 MVP를 줄이자고 제안한다.
- Should/Could 기능을 이유 없이 구현 범위에 넣지 않는다.
