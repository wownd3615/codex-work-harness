# Security Hardening

AI 코딩에서 보안은 “나중에 점검”이 아니라 기능을 만들 때 같이 보는 항목이다.

## 반드시 보는 경우
- 로그인, 회원가입, 세션, 토큰
- 관리자/권한/역할
- 결제, 구독, 환불
- 개인정보, 회사 내부 자료
- 파일 업로드/다운로드
- 외부 API, 웹훅, 자동 이메일/문자
- 배포, 환경변수, 서버 설정

## 기본 체크
- secret/API key가 client 코드나 공개 저장소에 없는가
- 권한 체크가 서버에서 되는가
- 입력값 검증이 서버에 있는가
- 파일 업로드는 확장자, MIME, 크기, 저장 위치 제한이 있는가
- 에러 메시지가 내부 경로, SQL, 토큰, 서버 정보를 노출하지 않는가
- 개인정보를 꼭 필요한 만큼만 저장하는가
- 로그에 비밀번호, 토큰, 주민번호, 계좌 같은 값이 남지 않는가
- CORS, redirect URL, webhook signature가 너무 넓게 열려 있지 않은가
- dependency 추가가 보안 위험을 늘리지 않는가

## 금지
- 브라우저 코드에 secret 넣기
- “프론트에서 숨겼으니 안전하다”라고 판단하기
- 관리자 버튼만 숨기고 서버 권한 체크를 빼기
- 샘플 키, 테스트 토큰, 내부 URL을 커밋하기
- 파일 업로드를 검증 없이 저장하기

## 작업 방식
- 위험 기능은 먼저 위협을 3개 이하로 적는다.
- 보안 조치는 기능 구현과 같은 PR/작업에 포함한다.
- 확신이 없으면 “추정”이라고 말하고 공식 문서나 실제 코드로 확인한다.
- 보안 수정은 가능한 작게 나눠 검증한다.

## 보고
- 확인한 보안 항목
- 수정한 위험
- 아직 확인 못 한 위험
- 사용자가 직접 설정해야 하는 환경변수/권한
