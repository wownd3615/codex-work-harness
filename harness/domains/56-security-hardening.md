# Security Hardening

AI 코딩에서 보안은 “나중에 점검”이 아니라 기능을 만들 때 같이 보는 항목이다.

## 반드시 보는 경우
- 로그인, 회원가입, 세션, 토큰
- 관리자/권한/역할
- 결제, 구독, 환불
- 개인정보, 회사 내부 자료
- 파일 업로드/다운로드
- 외부 API, 웹훅, 자동 이메일/문자
- 배포, 환경변수, 서버 설정

## 보안 레벨
작업 시작 때 가장 높은 레벨 하나로 분류한다.

| 레벨 | 상황 | 필수 처리 |
|---|---|---|
| S0 | 로컬 실험, 공개 데이터, 저장 없음 | 기본 체크 |
| S1 | 내부용, 파일 읽기/쓰기, API key 없음 | 기본 체크 + 백업 |
| S2 | 로그인, 회사 파일, 개인정보, 외부 API, 파일 업로드 | Threat Model + `check-security.ps1` + Security Review |
| S3 | 결제, 관리자, 공개 서비스, 고객 데이터, 자동 전송/삭제, LLM tool 실행 | S2 + 단계적 배포 + 로그/알림 + 사람 검토 |

S3는 바이브코딩만으로 안전하다고 말하지 않는다. 전문가 검토나 운영 검증이 필요하다고 보고한다.

## 기본 체크
- secret/API key가 client 코드나 공개 저장소에 없는가
- 권한 체크가 서버에서, 매 요청마다, 객체 단위로 되는가
- 입력값 검증이 서버에 있는가
- 파일 업로드는 확장자, MIME, 크기, 저장 위치 제한이 있는가
- 에러 메시지가 내부 경로, SQL, 토큰, 서버 정보를 노출하지 않는가
- 개인정보를 꼭 필요한 만큼만 저장하는가
- 로그에 비밀번호, 토큰, 주민번호, 계좌 같은 값이 남지 않는가
- CORS, redirect URL, webhook signature가 너무 넓게 열려 있지 않은가
- dependency 추가가 보안 위험을 늘리지 않는가

## 작업 전 위협 모델
위험 기능은 코딩 전에 아래 4개를 짧게 적는다.

- 지켜야 할 데이터: 계정, 결제, 개인정보, 회사 파일, API key 등
- 공격자: 일반 사용자, 권한 없는 직원, 외부 공격자, 악성 파일 업로더 등
- 위험 행동: 몰래 보기, 수정, 삭제, 다운로드, 권한 상승, 비용 발생
- 막는 방법: 서버 권한 체크, 입력 검증, 제한, 로그, 승인 절차

`templates/threat-model-lite.md`를 사용한다.

## 중단 조건
아래 상황에서는 바로 구현만 진행하지 말고 먼저 확인한다.

- 사용자가 실제 고객 데이터, 결제, 회사 기밀을 다룬다.
- 삭제, 전송, 결제, 권한 변경이 자동 실행된다.
- 권한 모델이 불명확하다.
- 어떤 데이터가 외부 서비스로 나가는지 불명확하다.
- secret이 이미 커밋되었을 가능성이 있다.
- 보안 검증 방법을 만들 수 없다.

## 인증과 세션
- 비밀번호는 직접 저장하지 않는다. 검증된 인증 서비스나 안전한 해시를 사용한다.
- 관리자, 결제, 개인정보 화면은 가능하면 MFA를 둔다.
- 로그인 실패, 비밀번호 재설정, 세션 만료 정책을 확인한다.
- 세션 쿠키는 가능하면 `HttpOnly`, `Secure`, `SameSite`를 적용한다.
- 로그인/인증 API에는 rate limit 또는 lockout을 검토한다.
- cookie 기반 인증에서 상태 변경 요청은 CSRF 방어를 확인한다.

## 권한과 데이터 접근
- 기본값은 deny다. 허용할 이유가 있을 때만 허용한다.
- “로그인했는가”와 “이 데이터를 볼 권한이 있는가”를 분리한다.
- URL id, 파일 id, 주문 id, 고객 id는 사용자가 추측할 수 있다고 가정한다.
- 목록, 상세, 수정, 삭제, 다운로드 각각 서버 권한 체크를 둔다.
- 관리자 버튼을 숨기는 것은 보안이 아니다. 서버에서 막아야 한다.
- 권한 실패는 안전하게 종료하고 필요한 만큼만 로그로 남긴다.

## 입력, DB, 파일 경로
- SQL은 문자열 결합 대신 parameterized query, ORM의 안전한 바인딩, prepared statement를 사용한다.
- shell command, 파일 경로, URL, HTML에 사용자 입력을 직접 넣지 않는다.
- HTML 출력은 escape/sanitize 한다. `dangerouslySetInnerHTML` 같은 기능은 특별한 이유가 있을 때만 쓴다.
- 파일 경로는 allowlist와 base directory 안에서만 처리한다.
- 외부 URL 호출은 SSRF를 고려해 사설 IP, 내부 metadata 주소, 로컬호스트 접근을 제한한다.

## 파일 업로드/다운로드
- 허용 확장자 allowlist를 사용한다.
- MIME과 확장자만 믿지 말고 가능한 경우 파일 signature를 확인한다.
- 파일명은 사용자가 준 이름을 그대로 저장하지 않는다.
- 크기 제한, 개수 제한, 저장 위치 제한을 둔다.
- 가능하면 webroot 밖에 저장하고, 다운로드는 권한 확인 handler를 거친다.
- 업로드 후 처리 라이브러리와 변환기는 최신 상태를 유지한다.

## 로그와 모니터링
- 로그인 성공/실패, 권한 실패, 관리자 작업, 데이터 export, 파일 업로드, 설정 변경은 로그 후보로 본다.
- 로그에는 secret, token, 비밀번호, 주민번호, 카드번호 같은 민감값을 남기지 않는다.
- 운영 서비스는 심각한 보안 이벤트를 사람이 알 수 있는 방식으로 연결한다.

## dependency와 공급망
- 새 dependency는 꼭 필요한지 확인한다.
- lockfile을 유지한다.
- 가능한 경우 `npm audit`, `pip-audit`, `cargo audit`, `dotnet list package --vulnerable` 같은 생태계 도구를 사용한다.
- 알 수 없는 skill, MCP, 플러그인, 스크립트는 실행 전 출처와 권한을 확인한다.
- GitHub Actions, 배포 스크립트, Dockerfile, IaC 변경은 secret, 권한, 외부 action pinning을 확인한다.

## AI/LLM 기능
LLM, agent, 자동 실행 기능이 있으면 추가로 확인한다.

- 사용자 입력과 시스템 지시를 같은 신뢰도로 보지 않는다.
- LLM 출력은 코드, SQL, shell, HTML, 파일 경로로 바로 실행하지 않는다.
- 도구 호출, 파일 삭제, 결제, 이메일/문자 발송, 외부 전송은 권한과 승인 단계를 둔다.
- LLM에게 secret, 내부 파일, 개인정보를 필요 이상으로 넘기지 않는다.
- prompt injection, 민감정보 노출, 과도한 권한, 공급망 위험을 별도 위험으로 기록한다.

## 금지
- 브라우저 코드에 secret 넣기
- “프론트에서 숨겼으니 안전하다”라고 판단하기
- 관리자 버튼만 숨기고 서버 권한 체크를 빼기
- 샘플 키, 테스트 토큰, 내부 URL을 커밋하기
- 파일 업로드를 검증 없이 저장하기
- LLM 출력이나 사용자 입력을 검증 없이 실행하기
- SQL, shell, HTML, 파일 경로에 사용자 입력을 문자열 결합으로 넣기

## 작업 방식
- 위험 기능은 먼저 위협을 3개 이하로 적는다.
- `scripts/check-security.ps1`로 기본 secret/파일 위험 스캔을 돌린다.
- 보안 조치는 기능 구현과 같은 PR/작업에 포함한다.
- 확신이 없으면 “추정”이라고 말하고 공식 문서나 실제 코드로 확인한다.
- 보안 수정은 가능한 작게 나눠 검증한다.
- S2/S3 작업은 가능하면 Security Specialist 또는 별도 리뷰어가 읽기 전용으로 먼저 본다.
