# Consistency System

코드와 디자인은 프로젝트 전체에서 한 사람이 만든 것처럼 보여야 한다.

비전공자식으로 말하면, 버튼/색상/폴더/함수 이름이 매번 달라지면 나중에 고치기 어려운 집이 된다.

## 핵심 원칙
- 새로 만들기 전에 기존 코드를 먼저 찾는다.
- 기존 컴포넌트, 스타일, 함수, 폴더 구조가 있으면 우선 재사용한다.
- 새 색상, 새 폰트, 새 간격, 새 버튼 모양을 이유 없이 만들지 않는다.
- 같은 역할의 UI는 같은 컴포넌트와 같은 상태 표현을 쓴다.
- 코드 분리는 유지보수에 도움이 될 때만 한다. 과한 분리는 금지한다.
- 중복 코드나 중복 스타일이 생기면 작업 끝에 정리한다.

## 디자인 기준 파일
사용자 화면이 있는 프로젝트는 가능하면 루트에 `DESIGN.md`를 둔다.

`templates/design-system-lite.md`를 복사해서 시작한다.

`DESIGN.md`에 적을 것:
- 브랜드/서비스 분위기
- 색상 토큰
- 폰트와 글자 크기
- 간격과 레이아웃 기준
- 버튼, 카드, 입력창, 테이블 기준
- 로딩, 빈 상태, 에러 상태
- 금지할 디자인 습관

## 코드 기준 확인
수정 전 아래를 먼저 찾는다.

- 비슷한 기능을 하는 함수
- 비슷한 화면이나 컴포넌트
- API 호출 방식
- 상태 관리 방식
- 폴더/파일 이름 규칙
- 테스트 작성 방식

찾은 뒤에는 기존 방식에 맞춘다. 새 방식을 만들려면 이유를 보고한다.

## UI 통일 체크
새 화면이나 컴포넌트를 만들 때 확인한다.

- 기존 버튼과 높이, radius, hover, disabled 상태가 같은가
- 카드/모달/입력창의 여백과 테두리가 같은가
- 색상이 `DESIGN.md` 또는 기존 토큰에서 왔는가
- 모바일에서 글자, 버튼, 표가 겹치지 않는가
- 비슷한 컴포넌트를 새로 만들지 않았는가
- 빈 상태, 로딩, 에러 상태가 같은 언어로 표현되는가

## 금지
- 같은 색상을 새 hex 값으로 계속 추가하기
- 페이지마다 버튼 모양을 다르게 만들기
- 기존 컴포넌트가 있는데 새 컴포넌트 만들기
- CSS를 임시로 덧붙이고 정리하지 않기
- UI 라이브러리나 아이콘 라이브러리를 이유 없이 추가하기
- “고급스럽게”라는 말만 믿고 장식용 효과를 늘리기

## 완료 보고
UI나 코드 구조를 바꿨다면 보고에 포함한다.

- 재사용한 기존 컴포넌트/패턴
- 새로 만든 컴포넌트/스타일과 이유
- 삭제한 중복 코드/스타일
- 확인한 화면 크기 또는 테스트
