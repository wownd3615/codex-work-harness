# Code Quality Gates

코드 품질은 “멋진 구조”가 아니라 나중에 고치기 쉬운 상태를 뜻한다.

## 작업 전
- 비슷한 기능, 함수, 컴포넌트, API 호출을 먼저 검색한다.
- 기존 폴더 구조와 파일 이름 규칙을 따른다.
- 테스트 방식과 에러 처리 방식을 확인한다.
- 새 라이브러리가 꼭 필요한지 먼저 판단한다.

## 구현 중
- 요구를 만족하는 가장 짧은 코드를 쓴다.
- 한 함수/파일이 너무 많은 책임을 가지면 역할별로 나눈다.
- 반대로 한 번만 쓰는 작은 코드를 과하게 분리하지 않는다.
- 이름은 역할이 드러나게 짓는다.
- 숨은 전역 상태, 임시 플래그, 복붙 코드를 피한다.
- 기존 패턴과 다른 방식을 쓰면 이유를 남긴다.

## 작업 후
- unused import, unused 함수, unused 변수 제거
- 새 코드 때문에 대체된 예전 코드 삭제
- 같은 로직이 2곳 이상 생겼는지 확인
- 타입/린트/테스트/실행 중 가능한 검증 수행
- 검증하지 못한 것은 못 했다고 보고

## 리팩터링 기준
리팩터링은 아래 조건일 때만 한다.

- 중복 때문에 버그 위험이 커진다.
- 파일이 너무 커져 변경 위치를 찾기 어렵다.
- 같은 변경을 여러 곳에 반복해야 한다.
- 테스트나 검증이 가능하다.

금지:
- 기분상 예뻐 보이게 구조만 바꾸기
- 동작 변경과 대규모 정리를 한 번에 섞기
- 사용자 변경을 임의로 되돌리기

## 보고
- 재사용한 기존 패턴
- 새로 만든 구조와 이유
- 삭제한 불필요 코드
- 실행한 검증
- 남은 위험
